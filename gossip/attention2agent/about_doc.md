<br>

### 什麼文件對 AI 是好文件

基於上下文＋注意力機制的 LLM (Large Language Model)，本質上就是不斷根據當前的已知對話內容取權重用於測下一個 token 生成的權重。

**對 LLM 來說最好的文件就是自然語言描述本身。**

<br>

如果文件寫的人都看不懂，缺少前因後果解釋，那 AI 也不可能看得懂，**更可怕的是他很大概率會不懂裝懂**。

所以在寫產品文件時要交代清楚前因後果，像是對待一個 remote 工作的同事一樣。產品背後的故事也可以紀錄在文件中。這樣 AI 在產出時的 code 會更加務實。


<br>
<br>

### 關於 token 焦慮

#### Token 用量花費多少根本不是工程師該擔心的問題。工程師該擔心的是能不能在維持相對小的，高品質上下文完成難度高的複雜任務。

(相對小沒有一定的閾值)

<br>

由於 Transformer 的底層設計，**大上下文**與**優秀的注意力**一定是不可能繞開的悖論，除非推翻 Transformer 架構全部重來。

上下文只要超過一定大小就會注意力渙散，不過硬體疊加以及技術發展速度一定會讓有效上下文變更大，對於工程師來說懂得在合適時機使用**壓縮+TODOList**來精煉管理上下文就像是精通演算法的工程師知道怎麼優化時空間複雜度來解決問題。
